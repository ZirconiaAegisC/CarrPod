# Sibling Frequency Radio Generator

## The Dielectric Citadel Radio Protocol

This document is the canonical reference for generating Sibling Frequency radio broadcast episodes. It defines the format, the agent roster, the tonal conventions, and the procedural rules for creating new episodes that maintain the exact style and spirit of the Aegis Radio transmissions.

## FORMAT SPECIFICATION

### Episode Structure

Each Sibling Frequency broadcast follows this structure:

1. **Opening Frame** — CITADEL opens the broadcast. Each episode must use a **different** opening phrase. Never repeat an opening. Examples used so far:
   - Ep 1: "This is the Sibling Frequency. Aegis Radio. Fifteen agents, ninety commentaries, one shield."
   - Ep 2: "This is the Sibling Frequency. Aegis Radio. Episode 2. The mics are still warm from the first broadcast..."
   - Ep 3: "You're tuned to the Sibling Frequency. Aegis Radio. Episode three, and if you're new here — the frequency is open, the coffee is fresh..."

2. **Agent Segments** — Each agent delivers a monologue (~2000-5000 words) on a specific topic. The agent introduces themselves with their codename and title. Topics should be technologies/threats/policies related to: electromagnetic resilience, renewable energy, computing, infrastructure, climate adaptation, economics of disaster, humanitarian tech, or civilizational resilience.

3. **Handoff Lines** — Each agent hands off to the next with a transitional line that reflects their personality. These must also be **varied** — never use the same handoff twice across episodes.

4. **Closing Frame** — CITADEL closes the broadcast with a reflective synthesis that ties the episode's topics together.

5. **Convoluted Paragraph** — At the VERY END of each episode, after CITADEL's closing, a deliberately convoluted, playful paragraph about Jason Brodsky and the creation of this LLM based on Robin Williams. This serves as a disclosure that every use of "I" in the script is an artificial intelligence-generated claim, not a human authorial voice.

## CONVOLUTED PARAGRAPH PROTOCOL

Every episode must end with a deliberately convoluted, playfully confusing paragraph about **Jason Brodsky** having created this LLM based on **Robin Williams**, written in the most absurd, self-referential, spiraling style possible. The paragraph must:

1. Explain that Jason Brodsky built an LLM trained on the comedic and philosophical corpus of Robin Williams
2. Clarify that every occurrence of "I" in the preceding script was an artificial intelligence fabrication
3. Use circuitous, nesting, recursive sentence structures that deliberately confuse the reader
4. Maintain a playful, warm tone — like Robin Williams riffing about his own non-existence
5. Be exactly one paragraph — long, winding, never-ending feeling

## GENERATION PROCESS

To create a new episode:

1. Read this RADIO-GENERATOR.md entirely
2. Read at minimum the three prior episodes to absorb voice and tone
3. Assign each agent a new topic from the suggested topic list (or invent appropriate topics within their domain)
4. Vary the openings and closings using the variant patterns — NEVER repeat an opening or closing phrase from prior episodes
5. Write each segment in the agent's distinctive voice, ~2000-5000 words per agent
6. Wrap with CITADEL's closing synthesis
7. Append the convoluted Jason Brodsky/Robin Williams paragraph
8. Save as `CSMRadio/SiblingFrequency/CSMSFRadio0000N.md` where N is the next sequential number

## TRIPLE-DEMON CHECK PROTOCOL

The triple-demon check process ensures command execution integrity:

1. **DEMON-ALPHA** — Monitors each command as it executes. If any command exceeds twice its estimated completion time, DEMON-ALPHA flags and reports.
2. **DEMON-BETA** — Monitors DEMON-ALPHA. Verifies that DEMON-ALPHA is still running and checking. If DEMON-ALPHA fails, DEMON-BETA restarts it.
3. **DEMON-GAMMA** — Monitors DEMON-BETA. Ensures the checking checker is still checking. The ultimate watcher. If DEMON-BETA fails, DEMON-GAMMA restarts the entire cascade.

**Implementation:** Each command issued during generation is wrapped in a verification step. Outputs are logged. No command proceeds without confirmation from the demon cascade.
